25live-cleaner

A Python utility that turns raw 25Live Excel exports into analysis-ready data. Handles overnight events, stitches fragments together, and ships a full audit trail so the cleaning is reproducible.

Python pandas NumPy Tkinter JSON

▓ // the problem

25Live is the campus scheduling system used by most American universities. Every classroom booking, every event reservation, every facilities request lives in 25Live. When a department asks "how is this building actually being used?" or "are we double-booking the auditorium?", the answer is supposed to come from a 25Live export.

The export is broken in four specific ways, and every one of them silently corrupts the analysis if you do not catch it.

An overnight event, say a lock-in from 9pm Friday to 7am Saturday, is split across two rows. The Saturday row has the word cont where the start time should be. Any time-based analysis that runs on the raw export will throw the second row away or, worse, treat cont as a missing value and miss the overnight booking entirely.

A multi-day conference is split into one row per day, each with its own start and end time. The same event shows up three times. Utilization metrics get inflated.

The same building shows up as "Cox Science Building", "Cox Science Bldg", and "cox science building" depending on who entered the booking. Group-by analysis treats them as three different rooms.

And exporting the same date range twice from two different reports produces duplicates that look identical until you check the row count.

The standard fix for all of this is hours of manual cleaning in Excel. The cleaning is not documented. The next person who runs the analysis cannot reproduce it. Somebody has to redo the work every quarter.

▓ // the approach

The script does five things in order. Each one solves a specific failure mode in the raw export.

First, it reads the file and maps the columns to a known schema, so a user dropping in a slightly different export version still gets a working pipeline.

Second, it detects the cont tokens and converts them into proper midnight crossings, so an overnight event becomes a continuous time interval instead of two broken rows.

Third, it normalizes the text fields, so "Cox Science Bldg" and "cox science building" collapse to the same identifier before any grouping happens.

Fourth, it stitches together adjacent fragments of the same event into one continuous record, using a configurable tolerance for small gaps.

Fifth, it deduplicates, but with a safety check: if two seemingly-identical rows have different locations, that is actually a real signal (the same event in two rooms), not a duplicate, so both rows are kept.

And underneath all of it, the script writes a JSON audit log documenting every transformation and embedding a SHA-256 fingerprint of the script itself. Anyone running the same input through the same script version gets the same output, every time, with proof.

▓ // how it works, in plain english

Four code snippets from the actual script. Each one is followed by a plain-English explanation. No prior programming knowledge needed.

Detecting the cont token

// cleaner_25live.py 
def _is_cont(v) -> bool:
    if pd.isna(v):
        return False
    s = str(v).strip().lower()
    return s == "cont" or s.startswith("cont ")

This function asks a single question: is the value in this cell the word "cont"? It strips out any extra spaces, lowercases the text, and checks whether the cleaned-up text is either exactly "cont" or starts with "cont " (so "cont 22:00" still gets flagged). The function returns true or false, nothing else. This is the foundation: before you can fix overnight events, you have to identify them. Every row in the export gets passed through this check.

Stitching event fragments back together

// cleaner_25live.py · simplified excerpt 
# For each group of rows that share the same event,
# location, and organization, walk through them in time order.
for _, r in valid.iterrows():
    if open_start is None:
        open_start = r[start_col]
        open_end = r[end_col]
    else:
        # If this fragment starts within 1 minute of where the
        # previous one ended, treat it as a continuation.
        if (r[start_col] - open_end) <= pd.Timedelta("1min"):
            if r[end_col] > open_end:
                open_end = r[end_col]
        else:
            # Otherwise, close the current event and start a new one.
            stitched_rows.append(open_row)
            open_start = r[start_col]
            open_end = r[end_col]

This is the heart of the cleaning logic. Imagine you have a bunch of puzzle pieces of the same event, but the system gave them to you in scrambled fragments. The script walks through them in time order and asks, for each new fragment, "did the previous fragment end less than a minute before this one starts?" If yes, the two fragments belong to the same event, so the script extends the running event's end time. If no, the script closes out the current event and starts a fresh one. The result: a multi-day conference that came in as twelve fragments leaves the script as one continuous record with a single start time and a single end time. The one-minute tolerance is configurable, because not every institution defines "continuous" the same way.

Knowing when a duplicate is not actually a duplicate

// cleaner_25live.py · simplified excerpt 
# Group rows by everything EXCEPT location, then count how many
# distinct locations each group has.
master["__mixed_location"] = (
    master.groupby(cand_keys_ignore_loc, dropna=False)["__norm_Location"]
          .transform("nunique") > 1
)

# Drop a row only if it's a true duplicate AND the location matches.
# If the location differs, both rows are real and both are kept.
rows_to_drop = (
    (master["__dup_group_size"] > 1)
    & (~master["__mixed_location"])
    & (master["__dup_rank"] > 0)
)

This is the part most cleaning scripts get wrong. Naive deduplication just looks for rows that match on every field and throws out the copies. But in 25Live, the same event in two different rooms (a guest lecture broadcast simultaneously in two auditoriums, say) shows up as two near-identical rows that differ only in the location field. A naive deduplication would silently delete one of them. This snippet flags those cases first, by asking "for rows that match on everything except location, are there actually multiple distinct locations?" If yes, the rows are protected from deletion. Only true duplicates (same event, same location, same time, same organization) get dropped. The rest stay. This is the kind of edge case that takes weeks to find in production if you do not design for it from the start.

Proof that the cleaning is reproducible

// cleaner_25live.py · script identity capture 
def _script_identity():
    here = os.path.abspath(__file__)
    with open(here, "rb") as fh:
        sha256 = hashlib.sha256(fh.read()).hexdigest()
    return {
        "script_version": SCRIPT_VERSION,
        "script_sha256": sha256,
        "python_version": sys.version,
        "pandas_version": pd.__version__,
        "platform": platform.platform(),
    }

Every time the script runs, it takes a fingerprint of itself and embeds that fingerprint into the audit log. A SHA-256 hash is a 64-character code that uniquely identifies the contents of a file. If a single character of the script changes, the hash changes. This means anyone reviewing the cleaned data, six months or six years from now, can verify that the script that produced it has not been silently modified. Combined with the Python and pandas version numbers, this is the difference between "trust me, the data is clean" and "here is the exact version of the exact tool that produced this output, run on this exact platform." That distinction matters in any institution where the analysis might end up in front of an auditor, a journal reviewer, or a senior dean.

▓ // what i learned

The hardest part of this project was not the time-handling logic, even though that is the part that looks technically dense. The hardest part was the deduplication. Naive deduplication is one line of pandas. Correct deduplication requires understanding the institutional context well enough to know that two near-identical rows with different locations are real signal, not noise. That insight came from talking to the facilities staff who actually use 25Live, not from reading the documentation.

The audit trail with the SHA-256 fingerprint started as overkill and ended up being the most reused part of the codebase. Every analytical pipeline I have built since uses the same pattern. Knowing exactly which version of which script produced a given output is, in an institutional setting, the difference between trustworthy analysis and "we will have to redo this from scratch."

If I were to rewrite this today, I would make the column mapping configurable through a sidecar YAML file rather than hardcoded. 25Live exports drift over time as institutions reconfigure their schemas, and the script's biggest fragility is that it assumes a specific column layout. Everything else has held up.
